## 確定した設計方針

### 1. カードモードの再設計

**核心の気づき: 初心者カードと編集カードは概念が違う**

| | 初心者カード（層2） | 編集カード（層3） |
|--|--|--|
| メタファー | コルクボード | バインダー／台本 |
| 1枚の単位 | 1話（自由に書く） | 1話（構造を管理する） |
| 順番 | あとで決めていい | 章立てが決まっている |
| ステータス | なし（または 書いた／まだ） | draft/writing/done |
| プロジェクト概念 | なし | あり（.aep3proj） |
| 感情目標 | 「こんなに書けた！」 | 「本の形になってきた」 |

**初期画面: B+C方式**

```
┌─────────────────────────────────────────────┐
│                                             │
│   ┌──────────────────────┐                 │
│   │ 📖 サンプル           │                 │
│   │ ──────────────────── │                 │
│   │ 主人公が駅のホームで  │                 │
│   │ 見知らぬ手紙を拾う。  │                 │
│   │ 差出人の名前は───     │                 │
│   │                      │                 │
│   │  1,200字  ✏️ 下書き   │                 │
│   └──────────────────────┘                 │
│                                             │
│            + 自分の話を書く                 │
│                                             │
└─────────────────────────────────────────────┘
```

- サンプルカードは「書きかけ」にする（完成品だと「自分には無理」になる）
- 「サンプル」ラベルは小さく、でも明確に
- `+ 自分の話を書く` が唯一の明確な次のアクション

**初心者→編集への移行トリガー（すべて必要・段階的に実装）**

- 話数が増えたとき（例: 5話超えで「章分けしてみませんか？」）
- 並び替えが増えたとき（構造化の提案）
- EPUB変換のとき（「構成を確認しますか？」）
- ユーザーが自分で選ぶ（かんたんモード / 編集モード）

### 2. チートシートの再設計

**核心の気づき: 全種類必要・でもタイミングで出し分ける**

現在のE7-B（5タブの静的パネル）→ コンテキスト依存型に変更

```
いつ                  何を出す
────────────────────────────────────────────
初めて開いたとき       背中押し ＋ 操作ガイド
書いている最中         記法リファレンス（必要な時だけ）
手が止まったとき       背中押し
新機能に触れたとき     操作ガイド
```

**実装方針: ルールベース（状態検知ベースは将来課題）**

- 「初回起動時はこれ」「書き始めから5分はこれ」など条件を手で定義
- シンプルで予測可能・メンテナンスしやすい
- 状態検知ベース（カーソル位置・操作パターン分析）はテレメトリーが使える大規模アプリの手法

### 3. 楽しさの設計

**トーン原則: 「気づいたら嬉しい」じんわり型**

```
× 「見て見て！すごいでしょ！」（Duolingo的・主張が強い）
○ 「あ、気づいたら増えてた」（じんわり型・押しつけない）
```

**即時フィードバックの実装Tier**

| Tier | 要素 | 実装方針 |
|------|------|----------|
| 1（これがないとアプリじゃない） | リアルタイムプレビュー | 常時ON |
| 2（あると当たり前に感じる） | 文字数カウンター | 常時表示 |
| 3（あると嬉しい、なくても困らない） | 節目の通知 | OFF可能 |

**節目通知の実装イメージ**

```
10000字突破 → 派手なアニメーションではなく
              ステータスバーにそっと「🎉 10,000字」と出て
              2秒後に消える

連続執筆3日 → バッジではなく
              次回起動時に「3日連続ですね」とひとこと
```

### 4. 初回起動ウィザード

**方針: ウィザード廃止・画面自体で代替する**

ウィザードは「設計が足りない画面の言い訳」になりがち。
各画面を丁寧に設計することでウィザードを不要にする。

```
ウィザードの役割              代替する設計
──────────────────────────────────────────────
「何をするアプリか教える」   → URL入力欄が大きく鎮座（層1の入口）
「書く機能を知ってもらう」   → プレビュー隅の「✏️ 自分でも書いてみる →」
「書き方を教える」           → B+Cの初期画面（書きかけサンプルカード）
「記法を教える」             → コンテキスト依存チートシート
「頑張れと言う」             → じんわり型の節目通知
```

### 5. 層1→層2の導線

**「読む人が書く人に変わる瞬間」の設計**

感情の山の分析:

```
感情の高まり
  │
  │   ③ Kindleで見る（本当の達成感）← 現在はアプリの外
  │         ↑ Kindle送信機能があればここも設計できる【将来課題】
  │   ② プレビューで本の体裁を見る ← 設計できる起点（ここに種をまく）
  │
  └──────────────────────────────────→ 時間
    URL貼る  変換  プレビュー  Kindle
```

**実装: プレビュー画面の隅に小さなリンク**

```
変換直後のプレビュー画面
  右下あたりに小さく
  「✏️ 自分でも書いてみる →」

  主張しない。でも見える。気づいた人だけでいい。
  かっこよさより確実に届くことを選ぶ。
```

## 設計全体を貫く原則（このセッションで発見）

**「機能が見えないと無いと思って使わない」×「多すぎるとどうしていいか分からない」**

→ まずこれ、が明確で、その先があることが見える設計

**「かっこよさは求めない」**

→ 洗練より、ちゃんとユーザーに届くことを優先する

**「全部必要・でも同時に出さない」**

→ 優先順位ではなく「いつ出すか」の問題として解く
