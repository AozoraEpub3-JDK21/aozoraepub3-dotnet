# AozoraEpub3-dotnet モダン執筆環境統合プロジェクト 要件・仕様まとめ

## 1. プロジェクト概要

Java版「AozoraEpub3」の .NET 移植プロジェクト（AozoraEpub3-dotnet）において、単なる変換ツールではなく、**「Web小説家や技術系同人作家がストレスなく執筆でき、即座に縦書きEPUBのプレビューを確認できる統合執筆環境」**を構築する。
現在、完成したEPUBのプレビュー機能自体は実装済みであり、本計画では「入力（エディタ）〜リアルタイム変換〜執筆中プレビュー」のフロントエンドおよび変換ロジックの設計・実装をスコープとする。

## 2. コアとなる設計思想（UX/UI）

* **「書くこと」への極限の集中（秀丸・サクラエディタの哲学）：**
執筆を行うメインエディタは、重いIDEのようなもっさり感を完全に排除する。ゼロ遅延のタイピング体験を最優先し、レンダリングなどの重い処理はエディタのUIスレッドを絶対にブロックしない設計とする。
* **プレビューの完全分離：**
執筆画面とプレビュー画面は「別ウィンドウ」とする。マルチディスプレイ環境などで、書きながら横目で縦書きのレンダリング結果（WebView2）を確認できるスタイルを想定。
* **執筆と装飾（CSS設定）のフェーズ分離：**
「書きながらCSSテンプレートをいじる」機能は持たせない。執筆中は最低限の縦書き表示にとどめ、CSSの詳細な調整は「原稿が仕上がった後の書き出し設定」として明確にワークフローを分ける。

## 3. 入力仕様と変換ロジック（フェーズ1）

Markdownの構造化の強みと、Web小説投稿サイト（なろう・カクヨム）の日本語特有の記法の打ちやすさを組み合わせた**「ハイブリッド仕様」**を採用する。
読者側のHTMLをスクレイピングする `narou.rb` とは異なり、執筆者の「生の入力テキスト」をリアルタイムで「青空文庫記法」へ置換するアプローチをとる。

### 3-1. 変換マッピング（正規表現ベース）

C# 11の `[GeneratedRegex]` を活用し、コンパイル時生成による爆速の文字列置換エンジンを構築する。

| 機能 | エディタ上の入力仕様 | 青空文庫記法（内部変換後） |
| --- | --- | --- |
| **ルビ** | `|漢字《かんじ》` (半角パイプ) | `｜漢字《かんじ》` (全角パイプ) |
| **傍点** | `《《強調》》` (カクヨム式) | `［＃傍点］強調［＃傍点終わり］` |
| **太字** | `**重要**` (Markdown式) | `［＃太字］重要［＃太字終わり］` |
| **見出し** | `# 第1章`, `## 1-1` (Markdown式) | `［＃大見出し］...`, `［＃中見出し］...` |
| **改ページ** | `---` (Markdown式) | `［＃改ページ］` |
| **字下げ** | `> 引用` (Markdown式) | `［＃ここから１字下げ］...` |

### 3-2. 小説向け自動整形（Lint / Formatter）機能

執筆者の負担を減らすため、変換エンジンを通す前に以下の自動補正処理（クリーニング）を挟む。

* `...` や `。。。` → `……`（三点リーダー）への変換
* `--` → `――`（ダッシュ2倍）への変換
* `！？` などの直後に全角スペースを自動挿入（文末除く）

## 4. エディタUIとプレビュー連動機構（フェーズ2）

### 4-1. エディタUI

* **モード選択:** 画面上部にコンボボックスを配置し、ルビなどの挙動を変えるためのモード（なろう、カクヨム等）を選択可能にする。
* **CMSライクなツールバー:** 記法を覚えていなくてもクリックやショートカット（例: `Ctrl+R`）で「ルビ」「傍点」などのタグを挿入できるボタンを配置。タグ挿入後は自動的に入力箇所へキャレットを移動させる実用的なUXとする。

### 4-2. 軽量リアルタイム・プレビュー機構

* **デバウンス（Debounce）処理:** テキスト入力イベント（`TextChanged`）に対して300ms〜500msの遅延判定を入れ、タイピングが一時停止したタイミングでのみ裏側の変換処理をキックする。
* **DOMの差分更新:** 別ウィンドウにある WebView2 への反映は、画面全体の再読み込み（チラつきの原因）を避け、`ExecuteScriptAsync` を用いて JavaScript 経由で `document.body.innerHTML`（または特定のコンテナ）のみを差し替える。

## 5. 新たな気づき・今後の検討課題（アーキテクチャ設計要件）

本構想を実現するにあたり、以下の技術的課題も併せて設計に組み込む必要がある。

1. **WPF/WinUI 3 での MVVMパターンの徹底:**
UIと変換ロジックが密結合になると今後の拡張（対応記法の追加など）が困難になるため、エディタのViewModel、変換エンジンのモデル、プレビューウィンドウへのメッセージング（Event Aggregator等）を明確に分離すること。
2. **エディタコントロールの選定:**
標準の `TextBox` では、将来的に「Markdown記法やルビ記法へのシンタックスハイライト（色付け）」を行いたくなった際に限界が来る。MVP（初期リリース）としては `TextBox` で進めるが、アーキテクチャ上は `AvalonEdit` などの高機能テキストコントロールへ差し替え可能な設計にしておくこと。
3. **変換ルールの外部化（ユーザー拡張性）:**
正規表現のルールをソースコードにハードコードするだけでなく、将来的に JSON や YAML 形式で外部から「カスタム変換辞書」を読み込める Strategy パターンでの実装を前提とする。

## 6. Claudeへの依頼事項

上記の要件定義に基づき、このプロジェクトをアジャイルに開発していくための「マイルストーン（フェーズ分け）」と、最初のフェーズで着手すべき具体的な「タスク一覧（クラス設計レベルの粒度）」を提案してください。
