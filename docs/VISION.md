## このアプリは何か

**「読む人」を「作る人」に変えるアプリ。**

なろうやカクヨムの小説をEPUBで読んでいた人が、
「自分でも書いてみたい」と思ったとき、
このアプリが書いて、本にして、届けるまでを支える。

ガチガチの小説家にはScrivenerやVellumがある。
でもそうじゃない人 — 初めて書く人、趣味で書く人、
同人誌を作りたい人 — が**楽しんで**創作できる環境。
それがこのアプリの存在理由。

## ユーザージャーニー（このアプリの本質）

```
Phase 0: 読者
  「好きな作品をEPUBで読みたい」
  → なろう/カクヨムのURLをアプリに貼る → EPUB完成
  → Kindle/Kobo/スマホで快適に読む
  （Java版AozoraEpub3の主要ユースケース。ここが入口）

      ↓ ある日、思う

Phase 1: 興味
  「自分でも何か書いてみたいかも」
  → でも本を作るって難しそう…
  → 専用ソフトは高いし、使いこなせなさそう…

      ↓ このアプリが背中を押す

Phase 2: 最初の一歩
  「あ、いつものアプリで書けるんだ」
  → 執筆モードを開く → テンプレートから始める
  → なろうで使い慣れた記法がそのまま使える
  → 右側にリアルタイムで縦書きプレビューが出る！

      ↓ 書いてみたら楽しかった

Phase 3: 本になる瞬間
  「これ、本にできるの？」
  → 「EPUB変換」ボタンを押す
  → 自分の書いた文章が、本のかたちになった
  → 「自分にもできた！」

      ↓ もっとちゃんと作りたくなる

Phase 4: 作り手として成長
  → 表紙を設定してみる
  → 目次を整える
  → テンプレートで見た目を変える
  → KindleやBOOK☆WALKERに出してみる
```

**重要なこと:**
- Phase 0（読者としてのEPUB変換）はこのアプリの**入口**であり**最大の差別化**
- どの競合も「既存の投稿サイトから作品を取り込んでEPUBにする」機能を持っていない
- この入口があるからこそ「読む→書く」の自然な移行が起きる

## 核となる価値

### 体験の転換
> 「自分には本なんて作れない」 → 「自分で本が作れた！」

この転換を起こすために必要なのは:
- 全部入りの巨大アプリではない
- 「書く → 本になる」の道筋が**見える**こと
- その道筋が**楽しい**こと

### 「1000人がこの機能だけは最高と言う」もの
個別の機能ではなく、**日本語で本を作る体験そのもの**。

それを支える3つの柱:
1. **EPUB変換の品質** — エンジン（青空文庫変換の蓄積）
2. **縦書きリアルタイムプレビュー** — 体験（書いた瞬間に本の姿が見える）
3. **日本語の本を作ることに特化した統合体験** — 約束（これ一つで完結）

| 属性 | 説明 |

## 競合と差別化

### ポジショニング

```
                 執筆機能の充実度
                      ↑
       Scrivener ●   │   ● Nola
        (プロ向け)    │   (書く特化)
                     │
         ──────────★┼──────────── 制作・出力の品質
           我々の位置  │
         (楽しく作る) │   ● Vellum
                     │   (出力特化)
                     ↓
```

### なぜ勝てるのか

| 強み | 競合にないもの |
|------|-------------|
| 読者→作者の導線 | URL→EPUB変換が入口（他にない） |
| 日本語ネイティブ | 縦書き、ルビ、投稿サイト記法 |
| 無料・OSS | Atticus $147、Vellum $249 |
| 全OS対応 | VellumはMac専用 |
| 楽しさ重視 | プロ向けの複雑さがない |

### 「なろう/カクヨムからEPUB」が最大の差別化であること

これは単なる変換機能ではない。**ユーザー獲得チャネル**。

Atticus/Vellumは「原稿がある人」しかユーザーになれない。
我々は「まだ書いていない人」=読者をユーザーにできる。
読者のほうが圧倒的に多い。
その読者の中から「書く人」が生まれる場所を作る。

## アプリの3つの層

```
層1: 読む＆変換 ─── なろう/カクヨム → EPUB（入口・差別化）
        ↓
層2: 書く ─────── 執筆モード（楽しく書ける環境）
        ↓
層3: 本にする ─── 編集＆制作（表紙、目次、テンプレート → EPUB出力）
```

**原則: 各層は独立して価値がある**
- 層1だけでも使える（Java版の既存ユーザー）
- 層2だけでも使える（エディタとして）
- 層3まで行くと「自分の本」ができる

**「小説」に絞らない:**
今の約束は小説だが、アーキテクチャは「本」を作るための汎用設計。
漫画、技術書、同人誌への拡張は将来の自然な成長。

## 設計原則

1. **楽しさファースト**
   創作の喜びを最大化する。機能の多さより、
   「書いた→本になった！」の感動が大事。

2. **書くことへの集中**
   執筆中にUIが邪魔をしない。重い処理はバックグラウンドで。

3. **段階的な複雑さ（Progressive Disclosure）**
   初心者は「書く→変換」の2ステップで完結。
   上級者は編集・制作で細部を制御。
   機能は使う人が増えるにつれて見えてくる。

4. **日本語ファースト**
   縦書き、ルビ、投稿サイト記法。
   英語圏ツールの翻訳ではなく、日本語の本のために設計する。

5. **入口を大切にする**
   URL→EPUB変換は「古い機能」ではなく「ユーザーとの最初の接点」。
   この入口の品質と体験を維持・向上し続ける。

6. **オフラインで完結**
   ネット接続なしで全機能が使える（AI・クラウド同期を除く）。

7. **変換品質への執着**
   AozoraEpub3の変換エンジンが持つ品質を維持・向上する。
   epubcheck合格を標準とする。

## 開発哲学

### 人はシステムになってはいけない

> 「仕事は楽しむべき。何ならゲーム的であるべき。
>  人はシステムを使うことはあっても、システムになってはいけない。
>  人がシステムになるなら、代わりのアプリを作ればいい。」

この原則は、このアプリのすべての設計判断に通底する。

**作者は「データ入力オペレーター」ではない。**
設定項目を正しく埋め、記法を正しく覚え、手順通りに変換ボタンを押す。
それは「システムの一部として正しく動く」ことであり、創作ではない。

**このアプリがすべきこと:**
- 作者が「書くこと」に没頭できる環境を作る
- 「本になった！」の感動を最短距離で届ける
- 使うほど楽しくなる仕掛けを入れる
- 正しい手順を強制するのではなく、自然に良い結果になるようにする

**このアプリがしてはいけないこと:**
- 設定項目で圧倒する（正しく設定することが「作業」になる）
- 記法の暗記を前提にする（学習が「義務」になる）
- 手順を間違えるとEPUBが壊れる（正解が1つしかない）
- 作者を「変換パイプラインの入力係」にする

### 楽しさのデザイン

「楽しい」は偶然生まれない。設計するもの。

**楽しさの3要素（このアプリにおいて）:**

1. **即時フィードバック — 書いた瞬間に反応がある**
   - リアルタイム縦書きプレビュー（書いた文字がすぐ本の姿になる）
   - 文字数カウンターがリアルタイムに増える
   - 入力した記法がプレビューに即座に反映される
   → ゲームでいう「ボタンを押したら即座に反応する」快感

2. **達成感の可視化 — やったことが目に見える**
   - 文字数の積み上げ（今日は1000文字書いた！）
   - マイルストーン達成（10000文字突破、初めての改ページ、初めてのEPUB変換）
   - プロジェクト全体の進捗バー
   → ゲームでいう「経験値バー」「実績解除」

3. **発見と探索 — 使うほど新しいことができる**
   - 最初はただ書くだけ → ルビを覚える → 見出しを使う → テンプレートを変える
   - 機能は最初から全部見せない。使い続けると自然に出会う
   - チートシートは「攻略ガイド」のように、必要な時に見る
   → ゲームでいう「新しいステージが解放される」感覚

### ゲーミフィケーション要素の具体案

**すぐ実装できるもの:**
| 要素 | 実装 | 楽しさの種類 |
|------|------|-------------|
| 今日の文字数 | ステータスバーに表示 | 即時フィードバック |
| 執筆セッション時間 | タイマー（任意） | 達成感 |
| 作品の総文字数 | プロジェクト情報 | 達成感 |

**中期で実装したいもの:**
| 要素 | 実装 | 楽しさの種類 |
|------|------|-------------|
| 執筆ストリーク | 連続執筆日数のカウント | 習慣化 |
| マイルストーン通知 | 「10000文字突破！」のトースト | 達成感 |
| 文字数グラフ | 日別の執筆量チャート | 可視化 |
| 目標設定 | 「今日は2000文字」→ プログレスバー | 達成感 |

**将来検討:**
| 要素 | 実装 | 楽しさの種類 |
|------|------|-------------|
| 初回体験バッジ | 「初めてのルビ」「初めてのEPUB」 | 発見 |
| 集中モード | 背景が変化、タイマー付き | 没入 |
| 作品統計ダッシュボード | 登場人物の出現頻度、章ごとの文字数分布 | 発見 |

**注意: 楽しさの押し付けはしない**
- すべてのゲーミフィケーション要素はOFF可能
- 「書くことに集中したい」人の邪魔をしない
- 通知は控えめに、作業を中断させない
- 数字に追われる感覚を作らない（目標はあくまで任意）

### ソフトウェア開発者の役割について

> 「Claudeのような高機能なプログラマーが登場したこれからは、
>  ソフトウェア開発者はただのプログラムする装置ではなく、
>  ユーザーがどのように考え何が楽しいかをメインに企画する方へ
>  移行すべき。」

このプロジェクト自体が、その実践である。

- **コードはAI（Claude）と協業して書く** — 実装の生産性を最大化
- **人間（開発者）は体験の設計に集中する** — 何を作るか、なぜ作るか
- **ユーザーの感情を設計する** — 機能リストではなく、喜び・驚き・達成感のリスト
- **「正しく動く」の先にある「楽しく使える」を追求する**

今日のプレビュー問題の修正（背景色のCSS 2行の追加）は、
AI（Claude）が5分で特定・修正できた。
一方、「このアプリは何者か」「誰のために何を届けるのか」の議論には
何時間もかかり、まだ続いている。
それこそが人間の開発者がやるべき仕事。
